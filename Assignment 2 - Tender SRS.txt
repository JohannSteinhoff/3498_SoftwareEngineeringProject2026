1. INTRODUCTION
1.1 Purpose of the SRS
This Software Requirements Specification (SRS) document formally defines the functional and non-functional requirements for Tender, a swipe-based recipe discovery and meal planning web application. The document is structured in accordance with IEEE 830-1998 standards and serves as the primary technical agreement between the development team, course instructors, and any stakeholders evaluating the project.


The SRS describes what the system does, not how it does it internally, and covers every user-facing feature already implemented in the working prototype as of February 2026. It also captures design constraints, non-functional expectations, and references used during development.
1.2 Problem Statement
Deciding what to cook is a daily challenge for individuals and families. Traditional recipe websites overwhelm users with too many choices, present recipes in a bland browsing interface, and fail to connect discovery with practical meal logistics. There is no single tool that:


* Makes recipe discovery fast and fun
* Connects discovered recipes directly to weekly meal planning
* Automatically propagates meal plans into an actionable grocery list
* Stores user dietary restrictions and preferences to personalize results


Tender addresses all four gaps by combining a swipe-based discovery UI (familiar from mobile dating applications), a weekly meal planner, and a smart grocery list — all behind a single authenticated account.
1.3 Intended Audience and Document Conventions
Intended Audience:


* Course instructor (primary evaluator)
* Development team members (implementation reference)
* Future contributors or maintainers
* People looking for recipes (People who don’t know what to make for dinner)




Document Conventions:


* FR-XX prefixes denote Functional Requirements.
* NFR-XX prefixes denote Non-Functional Requirements.
* C-XX prefixes denote Constraints.
* All feature descriptions are grounded in the actual source code files: tender.html, login.html, signup.html, dashboard.html, api.js, and server.js.
* Diagrams are presented in text/ASCII form; the team should replace them with properly rendered UML diagrams in the final PDF submission.
1.4 Product Scope
Tender is a browser-based, full-stack web application with the following scope:


In Scope
	Out of Scope
	User registration & login
	OAuth / social login (buttons exist but are non-functional placeholders)
	Swipe-based recipe discovery
	Real-time collaboration / family sharing
	Recipe CRUD (user-created recipes)
	External recipe API integration (Spoonacular, etc.)
	Weekly meal plan calendar
	Nutritional data / calorie tracking
	Interactive grocery list
	Mobile native app (iOS/Android)
	User profile with dietary/cuisine preferences
	Push notifications
	Admin user management
	Payment processing
	Responsive design (mobile + desktop)
	Image file upload (emoji icons used instead)
	Dark mode support
	

	

The application comprises five HTML pages (tender.html, login.html, signup.html, dashboard.html, index.html) and a REST API backend (server.js) backed by a SQLite database (tender.db). The frontend uses vanilla HTML5, CSS3, and JavaScript ES6+. The backend uses Node.js with the Express.js framework.

2. STATEMENT OF FUNCTIONAL REQUIREMENTS
2.1 User Registration
Description: New users can create an account through a multi-step wizard that collects account credentials and food preferences.


Trigger: User clicks "Create Account" on the landing page or "Create one" on the login page.


Preconditions: User is not already logged in (enforced by session check on all pages).


Functional Requirements:


ID
	Requirement
	FR-01
	The system shall provide a 3-step registration wizard. Step 1 collects first name, last name, email address, password, and password confirmation.
	FR-02
	The system shall validate that the password is at least 8 characters, contains one uppercase letter, and contains one number. Real-time visual indicators (checkmarks) shall update as the user types.
	FR-03
	The system shall validate that the two password fields match before allowing progression to step 2.
	FR-04
	The system shall validate the email address format using a regular expression.
	FR-05
	Step 2 shall collect cooking skill level (Beginner, Intermediate, Advanced, Pro Chef), household size (1, 2, 3–4, 5+), dietary restrictions (Vegetarian, Vegan, Gluten-Free, Dairy-Free, Keto, Halal, Kosher), and weekly food budget (Budget / Moderate / Flexible / Premium).
	FR-06
	Step 3 shall display 12 cuisine options (Italian, Mexican, Chinese, Japanese, Indian, Thai, Mediterranean, American, French, Korean, Greek, Vietnamese) and require the user to select at least 3 cuisines.
	FR-07
	Step 3 shall also collect how many meals the user cooks per week (1–3, 4–7, 8–14, 15+).
	FR-08
	Upon successful submission, the system shall call POST /api/auth/register, store the returned session token in localStorage under tender_token, and redirect to dashboard.html.
	FR-09
	If the email is already registered, the system shall display the error "This email is already registered. Please log in."
	FR-10
	A step progress indicator (3 dots) at the top of the form shall change state (pending / active / completed) as the user progresses through steps.
	

Post-condition: A new user record is created in the users database table. Associated dietary and cuisine preference records are written to user_dietary and user_cuisines tables.

2.2 User Authentication (Login / Logout)
Description: Registered users can securely log in to access protected features and log out to end their session.


Functional Requirements:


ID
	Requirement
	FR-11
	The login page (login.html) shall present an email field and password field.
	FR-12
	The system shall validate both fields on submit; empty or malformed entries shall show inline error messages.
	FR-13
	On successful login, the system shall call POST /api/auth/login, store the returned Bearer token in localStorage (tender_token), cache user data in localStorage (tender_user_cache), and redirect to dashboard.html.
	FR-14
	On failed login (wrong email/password), an error alert "Invalid email or password. Please try again." shall be shown for 5 seconds.
	FR-15
	The system shall redirect already-logged-in users away from login.html and signup.html to dashboard.html immediately on page load.
	FR-16
	A "Logout" button in the dashboard navigation shall call POST /api/auth/logout, remove tender_token and tender_user_cache from localStorage, and redirect the user to login.html.
	FR-17
	All pages in dashboard.html shall verify the session token on load by calling GET /api/auth/me; if the token is absent or expired, the user shall be redirected to login.html.

ID
	Requirement

3. NON-FUNCTIONAL REQUIREMENTS
3.1 Performance
ID
	Requirement
	NFR-01
	All page transitions within dashboard.html (switching between Home, Discover, Meal Plan, Grocery, Profile) shall occur within 100ms because they are pure client-side DOM show/hide operations with no network requests required to switch views.
	NFR-02
	The recipe discovery endpoint (GET /api/recipes/discover) shall return results within 500ms under normal load, as the SQLite database runs in-process with the Node.js server on the same machine.
	NFR-03
	The frontend shall load the initial seed recipe data from localStorage (tender_seed_recipes) on subsequent visits, avoiding redundant network fetches.
	NFR-04
	CSS animations (card hover transitions, modal fade-ins, button hover states) use hardware-accelerated CSS properties (transform, opacity) and are set to durations of 200–300ms to feel responsive without being distracting.
	NFR-05
	The dashboard.html file is a single-page application; all five views are loaded into the DOM at once, eliminating page load delays when switching between sections.
	3.2 Security
ID
	Requirement
	NFR-06
	User passwords shall never be stored in plain text. All passwords are hashed using bcryptjs before being written to the users database table.
	NFR-07
	Every protected API endpoint (recipes, grocery, meal plan, profile, admin) shall require a valid Bearer token in the Authorization header. Requests without a valid token shall receive HTTP 401 Unauthorized.
	NFR-08
	Session tokens are stored in localStorage under the key tender_token and are included in every API request via the Authorization: Bearer token header pattern in api.js.
	NFR-09
	Admin-only endpoints (/api/admin/*) shall verify the is_admin flag on the authenticated user's database record server-side; frontend-only role checks are not relied upon for security.
	NFR-10
	Input validation (email format, password strength) is performed on both the client side (real-time feedback in signup.html and login.html) and the server side (API returns 400 Bad Request for invalid inputs).
	NFR-11
	Users can only modify or delete their own recipes. The server verifies that recipe.created_by === authenticated_user.id before allowing PUT or DELETE operations; admins bypass this check.
	3.3 Usability
ID
	Requirement
	NFR-12
	The application shall be fully responsive. At or below 768px viewport width, the top navigation bar shall be replaced by a mobile bottom navigation bar.
	NFR-13
	The registration wizard shall display a visual step progress indicator (three dots) that shows the user's position in the 3-step form. Completed steps use a green color; the active step is highlighted in the primary brand color (#FF6B6B).
	NFR-14
	All form submission buttons shall show a disabled/loading state while an API request is in flight, preventing duplicate submissions.
	NFR-15
	Success and error feedback shall be shown via non-intrusive alert banners that auto-dismiss after 5 seconds.
	NFR-16
	The application shall support dark mode, applied automatically based on the user's OS-level prefers-color-scheme setting, and also persistable via the tender_theme localStorage key.
	NFR-17
	The Poppins font family is used consistently throughout all pages, loaded from Google Fonts with weights 300–800.
	NFR-18
	Empty states (no liked recipes, no grocery items, no meal plan entries) shall display a friendly icon and message rather than a blank space, guiding users toward the relevant action.
	NFR-19
	The recipe discovery cards shall display all relevant information (name, emoji, description, cook time, servings, difficulty, cuisine, ingredients, instructions) in a single view.
	3.4 Maintainability
ID
	Requirement
	NFR-20
	All API calls from the frontend are consolidated in a single client module (api.js) that exposes a TenderAPI object. This separation means backend URL or behavior changes only require edits in one file.
	NFR-21
	CSS design tokens (colors, gradients, border radii, shadows) are defined as CSS custom properties (variables) in a :root block. Dark mode overrides only the variable values, not individual component styles.
	NFR-22
	The dashboard is implemented as a single-page application using a switchPage() function that toggles the .active class on .page-view divs. Adding a new section requires adding one new .page-view div and one switchPage() call.
	NFR-23
	The backend follows a clear separation of concerns: server.js handles HTTP routing and middleware; database.js handles all SQL queries and data access; auth.js handles authentication helpers.
	NFR-24
	A TenderDB compatibility shim in api.js provides a backwards-compatible interface over the async TenderAPI calls, allowing older dashboard code to continue working while the codebase is incrementally modernized.
	NFR-25
	The project uses npm and a package.json for dependency management, making the development environment reproducible with a single npm install command.

4. DESIGN / IMPLEMENTATION CONSTRAINTS
4.1 Standards Compliance
ID
	Constraint
	C-01
	The application's HTML shall be written to the HTML5 standard, using semantic elements with proper lang and charset attributes on all pages.
	C-02
	The REST API shall follow RESTful design conventions: resources are named as nouns (/recipes, /grocery, /mealplan), HTTP verbs correctly communicate intent, and HTTP status codes 200, 201, 400, 401, 403, 404, 500 are used appropriately.
	C-03
	Passwords shall be hashed using bcrypt with a cost factor of 10 rounds (bcryptjs default), in compliance with OWASP password storage guidelines.
	C-04
	The SRS document is structured in accordance with IEEE 830-1998 — the standard for Software Requirements Specifications.
	C-05
	CSS layout uses CSS Grid and Flexbox, which are W3C standards supported in all modern browsers. No proprietary CSS extensions are used.
	4.2 Development Constraints
ID
	Constraint
	C-06
	Runtime: The backend requires Node.js v18 or higher. The project uses ES2017+ syntax (async/await) throughout server.js, database.js, and api.js.
	C-07
	Database: The project uses SQLite via the sql.js npm package (pure JavaScript, no native binary compilation required). The database is file-based (tender.db) and not suitable for multi-instance horizontal scaling without migration to a client-server database.
	C-08
	Session Storage: Authentication tokens are stored in localStorage (browser-side), which means sessions persist across browser tab close/reopen but are scoped to a single device and browser. Cross-device session sync is not supported.
	C-09
	No Dedicated File Storage: The application does not use a dedicated file storage backend or CDN for images (e.g., AWS, S3, Cloudinary). Recipe images uploaded by users are converted to Base64 data URIs on the client and stored as text directly within the database record. Recipes without uploaded images use emoji characters as visual identifiers.
	C-10
	Single-Process Server: The Node.js server runs as a single process with in-memory session data. A server restart clears all active sessions. Production deployment would require Redis or database-backed session storage.
	C-11
	Browser Dependency: The application requires a modern web browser (2020 or newer) with JavaScript enabled. It uses fetch(), async/await, CSS Grid, CSS Custom Properties, and localStorage.
	C-12
	Team Size and Scope: The project is scoped for a team of four students. The four principal feature areas (recipe discovery, meal planning, grocery list, user management) map naturally onto four developer responsibilities.
	C-13
	Platform: The development environment targets Windows (a start.bat quick-start script is included), though the Node.js stack is cross-platform and can run on macOS or Linux without modification.
